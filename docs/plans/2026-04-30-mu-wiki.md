# mu-wiki Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use devmuse:mu-code to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a new on-demand skill (mu-wiki) that generates and maintains project-level architecture documentation using a two-phase subagent approach.

**Architecture:** Two-phase generation (structure → pages) via inline Claude Code subagents. Integration with mu-scope (suggestion) and mu-arch (read + suggestion). Output to `docs/wiki/`.

**Tech Stack:** Markdown skill files, Claude Code Agent tool, git

**Note:** The scope lists `knowledge/templates/wiki-page.md` as a new file, but the design spec decided to inline page format requirements in SKILL.md instead (spec line 276). No separate wiki-page template is needed.

### Task 2: Create mu-wiki SKILL.md — core skill definition

**Covers:** UC-1, UC-2, UC-E1, UC-E2, UC-E3, UC-ERR1, UC-ERR2

**Files:**
- Create: `skills/mu-wiki/SKILL.md`

This is the core deliverable. The SKILL.md must define:

- [ ] **Step 1: Write the skill header and overview**

Include: name, description, skill type (on-demand), two modes (generate/update), anti-patterns.

- [ ] **Step 2: Write the generate flow checklist**

Steps in order:
1. Check if wiki already exists (UC-E3: prompt "覆盖重建还是增量 update？")
2. Size gate: `git ls-files | xargs wc -l | tail -1` — apply thresholds (<50k full, 50k-200k top-level only, >200k limit to top-level modules)
3. Collect signals: file tree (`git ls-files`), README.md, CLAUDE.md if exists
4. Dispatch Structure subagent (Explore type) with prompt template — output: wiki structure JSON
5. Present structure to user for review (show sections + pages + relevant_files)
6. User confirms or adjusts → loop back to step 4 if adjusted
7. Dispatch Page subagents (general-purpose, parallel) — one per page, each with page spec + relevant_files paths
8. Check subagent results — mark failed pages in _index.md (UC-ERR1)
9. Assemble _index.md from template at `@../../knowledge/templates/wiki-index.md`
10. Present output for user review → commit

- [ ] **Step 3: Write the Structure subagent prompt template**

Inline in SKILL.md. The prompt must instruct the subagent to:
- Read the provided file tree and README
- Analyze project structure and purpose
- Output JSON conforming to the structure schema (title, description, sections, pages with id/title/description/importance/relevant_files/related_pages)
- Produce 8-12 pages (comprehensive) or 4-6 pages (concise if `--concise` flag)
- relevant_files must be actual paths from the file tree
- Language follows user preference

Include the full JSON schema in the prompt.

- [ ] **Step 4: Write the Page subagent prompt template**

Inline in SKILL.md. The prompt must instruct each page subagent to:
- Read ALL files listed in relevant_files using Read tool
- Generate markdown: H1 title, intro (1-2 paragraphs), H2/H3 sections
- Include Mermaid diagrams (graph TD only, sequenceDiagram with proper syntax)
- Include Markdown tables for structured info
- Start each page with a `<details>` block listing ALL referenced source files
- Mandatory source citations: `Sources: [filename:start_line-end_line]()`, minimum 5 distinct files
- All info solely from source files, no fabrication
- Write output to `docs/wiki/<page-id>.md`
- Language follows user preference

- [ ] **Step 5: Write the update flow checklist**

Steps:
1. Read `docs/wiki/_index.md` — extract baseline commit
2. Run `git diff --name-only <baseline>..HEAD` — get changed files
3. Match changed files against each page's Relevant Files column in _index.md
4. Staleness check: if affected pages > 60% of total or changed files > 50, warn and suggest full regenerate (UC-E1)
5. If relevant_files no longer exist (renames/deletions), degrade to full regenerate (UC-ERR2)
6. Present affected pages to user — user confirms, adjusts, or cancels
7. Dispatch Page subagents for affected pages only (parallel)
8. Update _index.md: new baseline commit, history entry
9. User review → commit

- [ ] **Step 6: Write the process flow diagram**

Mermaid or DOT diagram covering both generate and update flows.

- [ ] **Step 7: Write key principles, integration section, and anti-rationalizations**

Integration section:
- Invoked by: direct `/mu-wiki` or suggestion from mu-scope/mu-arch
- Produces: `docs/wiki/` directory with _index.md + page files
- Terminal state: commit (no downstream skill invocation — mu-wiki is terminal)
- Template: `@../../knowledge/templates/wiki-index.md`

- [ ] **Step 8: Commit**

```bash
git add skills/mu-wiki/SKILL.md
git commit -m "feat(skills): add mu-wiki skill for architecture wiki generation"
```
