## Purpose

Support onboarding of a new contributor (Jeff) who just cloned the repo and wants a working mental model before considering contributions.

## Core Idea

DevMuse is a **Claude Code plugin** that bolts a disciplined software-engineering workflow onto Claude sessions. It is not an app, library, or service ŌĆö it is a *prompt-engineering artifact*: a bundle of markdown skills, markdown agents, one always-on bootstrap rule, two shell hooks, and domain-knowledge markdown, packaged as a plugin (`plugin.json`). At session start, a `SessionStart` hook injects `rules/bootstrap.md` so Claude always sees the skill-selection decision flow; thereafter the user's intent routes to a skill (`mu-biz`, `mu-prd`, `mu-scope`, `mu-arch`, `mu-plan`, `mu-code`, `mu-review`, `mu-debug`, `mu-retro`, `mu-write-skill`, plus the newly-added `mu-explore`) which the model follows like a checklist. Skills optionally dispatch one of two agents (`mu-reviewer` with six modes, `mu-coder`) and reference knowledge files on demand. The whole thing is content, not code.

## Component Map

Four-layer architecture (rules / skills / agents / knowledge) plus hooks and docs. Depth capped at 2 per onboarding rules.

- **`.claude-plugin/`** ŌĆö Plugin manifest(s) consumed by Claude Code marketplace/install.
  - `plugin.json` ŌĆö Declares skills directory and agent file list; `version: 0.2.0`.
  - `marketplace.json` ŌĆö Marketplace registration metadata.
- **`rules/`** ŌĆö Always-on principles, loaded via SessionStart hook (not plugin auto-discovery).
  - `bootstrap.md` ŌĆö The single rule. Decision flow for "which skill to invoke", red flags, priority ordering (user > skill > default).
- **`skills/`** ŌĆö 12 user-triggered workflow skills, each a directory with `SKILL.md` (and optionally scripts/ancillary docs).
  - Pipeline (auto-routed): `mu-scope` ŌåÆ `mu-arch` ŌåÆ `mu-plan` ŌåÆ `mu-code` ŌåÆ `mu-review`
  - Orthogonal (auto-routed): `mu-explore`, `mu-debug`, `mu-retro`
  - On-demand (slash-only): `mu-biz`, `mu-prd`
  - Router: `mu-route`
  - Meta: `mu-write-skill`
- **`agents/`** ŌĆö Two dispatchable sub-agents with isolated context.
  - `mu-reviewer.md` ŌĆö Six modes: review-design, review-plan, review-code, review-compliance, review-coverage, review-security.
  - `mu-coder.md` ŌĆö Implementation specialist invoked by `mu-code`.
- **`knowledge/`** ŌĆö On-demand reference material, pulled via `@../../knowledge/...` relative paths.
  - `languages/` ŌĆö Per-language review criteria (java, go, python, typescript).
  - `templates/` ŌĆö Artifact templates (scope.md, explore.md, architecture.md).
  - `principles/` ŌĆö Thinking rubrics (10 files: inversion, premise-check, chesterton's-fence, git-safety, stance-detection, sign-off-gate, architecture-assessment, graphviz-conventions, skill-cso, skill-testing).
  - `reviews/` ŌĆö Review checklists (security-checklist, design-audit-rubric).
- **`hooks/`** ŌĆö Shell hooks registered via `hooks.json`.
  - `session-start` ŌĆö Reads `rules/bootstrap.md` and injects as `additionalContext` on SessionStart (startup/clear/compact).
  - `pre-tool-use/pipeline-gate.sh` ŌĆö Before Edit/Write, refuses unless `docs/scope/*.md` and `docs/specs/*-design*.md` both exist; exempts edits inside the plugin dir; fail-open.
  - `pre-tool-use/destructive-guard.sh` ŌĆö Warns on destructive Bash commands.
- **`docs/`** ŌĆö Human-facing documentation and artifacts generated by previous runs of the skills.
  - `architecture.md` / `architecture_cn.md` ŌĆö Canonical architecture description (EN/CN).
  - `testing.md` / `testing_cn.md` ŌĆö Testing approach.
  - `scope/`, `specs/`, `plans/`, `proposals/` ŌĆö Accumulated artifacts from running scope / arch / plan / proposal workflows on devmuse itself (dogfooding).
- **`tests/`** ŌĆö Organized as scenario directories, not a unit-test tree.
  - `brainstorm-server/`, `claude-code/`, `explicit-skill-requests/`, `hooks/`, `skill-triggering/`, `subagent-driven-dev/` ŌĆö Each holds harness/scenario material for validating skill-invocation behavior. Not auto-run as part of plugin install.
- **`.github/`** ŌĆö Issue template directory and PR template.
- **`README.md` / `README_CN.md`** ŌĆö Bilingual top-level docs.

## Entry Points

Execution is not "program start" ŌĆö it is Claude Code session start. Entry points:

- `hooks/session-start` ŌĆö Fires on every SessionStart (startup, clear, compact). Injects `rules/bootstrap.md` into the system context. This is THE boot path.
- `hooks/pre-tool-use/pipeline-gate.sh` ŌĆö Fires before every `Edit` or `Write`. Gatekeeps the pipeline (scope + design must exist).
- `hooks/pre-tool-use/destructive-guard.sh` ŌĆö Fires before every `Bash`. Destructive-command guard.
- User utterance ŌåÆ `Skill` tool invocation ŌĆö User types `/mu-scope`, `/mu-arch`, etc., or says something that triggers the bootstrap decision flow; Claude invokes the Skill tool on the matching `skills/<name>/SKILL.md`.
- `.claude-plugin/plugin.json` ŌĆö Static manifest read at plugin install time; declares agents explicitly and `skills/` for auto-discovery.

## Key Flows

### Flow: Session boot (always runs)
1. User opens Claude Code session (or runs `/clear`, `/compact`).
2. Claude Code runs `hooks/session-start`.
3. Script reads `rules/bootstrap.md`, JSON-escapes it, emits `hookSpecificOutput.additionalContext`.
4. Claude's system context now includes the bootstrap rule ŌĆö every subsequent user message is filtered through the "which skill applies?" decision flow.

### Flow: Feature-work pipeline (happy path)
1. User: "add feature X" (or similar).
2. Bootstrap rule fires ŌåÆ Claude invokes `mu-scope` skill.
3. `mu-scope` produces Use Case Set ŌåÆ writes `docs/scope/<date>-<slug>.md`.
4. Claude invokes `mu-arch` ŌåÆ writes `docs/specs/<date>-<slug>-design.md`; dispatches `mu-reviewer` in `review-design` mode.
5. Claude invokes `mu-plan` ŌåÆ detailed implementation plan; optional `mu-reviewer` `review-plan`.
6. Claude invokes `mu-code` ŌåÆ sets up worktree, dispatches `mu-coder` for implementation, then `mu-reviewer` `review-compliance` + `review-code` between tasks.
7. Claude invokes `mu-review` ŌåÆ dispatches `mu-reviewer` `review-coverage` (+ optional `review-security`), handles feedback, finishes with merge/PR/keep/discard.

### Flow: Pipeline gate enforcement
1. Claude attempts `Edit` or `Write` on a non-plugin file.
2. `pipeline-gate.sh` receives tool input JSON on stdin.
3. Extracts `file_path`. If path is inside `CLAUDE_PLUGIN_ROOT`, exits 0 (exempt ŌĆö enables meta-work on devmuse itself).
4. Else checks `docs/scope/*.md` exists; if not, emits `permissionDecision: deny` with message "Run mu-scope first."
5. Else checks `docs/specs/*-design*.md` exists; if not, emits deny with "Run mu-design first."
6. Both present ŌåÆ exit 0 (allow). Any error ŌåÆ `trap 'exit 0' ERR` fail-open (never blocks due to bugs).

### Flow: Bug fix (short-circuit pipeline)
1. User: "fix this bug".
2. Bootstrap ŌåÆ `mu-scope` with 1 use case = repro steps.
3. `mu-debug` (4-phase systematic root cause).
4. `mu-code` for the fix.

## Domain Terms

| Term | Meaning |
|------|---------|
| `skill` | A markdown file under `skills/<name>/SKILL.md` with YAML frontmatter (`name`, `description`). Loaded via the `Skill` tool; acts as a workflow checklist Claude must follow. |
| `agent` | A markdown file under `agents/` declaring an independent sub-agent role with isolated context; invoked by skills via the Task tool. |
| `rule` | Always-on content from `rules/`, injected via SessionStart hook (not auto-discovered by plugin system). |
| `knowledge` | Passive reference content under `knowledge/`, included via `@`-path references inside skills/agents. |
| `UC / Use Case` | A scoped requirement unit produced by `mu-scope`; each UC has an ID (e.g., `UC-3`) that propagates through arch ŌåÆ plan ŌåÆ code ŌåÆ tests. |
| `UC-ID traceability` | Plan tasks, code comments, and tests all carry the UC-ID so coverage can be audited. |
| `mu-reviewer modes` | Six review stances selectable per invocation: review-design, review-plan, review-code, review-compliance, review-coverage, review-security. |
| `pipeline gate` | The pre-tool-use hook that refuses Edit/Write unless scope+design artifacts exist. Enforces the scopeŌåÆarchŌåÆplanŌåÆcodeŌåÆreview order. |
| `fail-open` | Hooks exit 0 on any internal error so plugin bugs never block the user. |
| `four categories` | Pipeline (scopeŌåÆarchŌåÆplanŌåÆcodeŌåÆreview, auto-routed) / Orthogonal (explore, debug, retro, auto-routed) / On-demand (mu-biz, mu-prd, slash-only) / Router (mu-route). |
| `quick mode` (of mu-biz) | 4 forcing questions instead of full business-model-canvas analysis. |
| `Quick Probe` | Rapid codebase impact scan inside `mu-scope` before formal UC enumeration. |
| `dogfooding` | devmuse uses its own skills on itself ŌĆö see `docs/scope/`, `docs/specs/`, `docs/plans/` populated with internal devmuse work. |
| `CLAUDE_PLUGIN_ROOT` | Environment var Claude Code sets so hooks can locate the plugin; used by both hook scripts and the plugin-self-edit exemption. |
| `Superpowers` | Upstream project by Jesse Vincent; devmuse is "based on" it. `gstack` (Garry Tan) contributed security/design-audit/premise/hook patterns. |
