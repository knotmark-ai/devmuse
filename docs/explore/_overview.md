## Purpose

Support onboarding of a new contributor (Jeff) who just cloned the repo and wants a working mental model before considering contributions.

## Core Idea

DevMuse is a **Claude Code plugin** that bolts a disciplined software-engineering workflow onto Claude sessions. It is not an app, library, or service тАФ it is a *prompt-engineering artifact*: a bundle of markdown skills, markdown agents, one always-on bootstrap rule, two shell hooks, and domain-knowledge markdown, packaged as a plugin (`plugin.json`). At session start, a `SessionStart` hook injects `rules/bootstrap.md` so Claude always sees the skill-selection decision flow; thereafter the user's intent routes to a skill (`mu-biz`, `mu-prd`, `mu-scope`, `mu-arch`, `mu-plan`, `mu-code`, `mu-review`, `mu-debug`, `mu-retro`, `mu-write-skill`, plus the newly-added `mu-explore`) which the model follows like a checklist. Skills optionally dispatch one of two agents (`mu-reviewer` with six modes, `mu-coder`) and reference knowledge files on demand. The whole thing is content, not code.

## Component Map

Four-layer architecture (rules / skills / agents / knowledge) plus hooks and docs. Depth capped at 2 per onboarding rules.

- **`.claude-plugin/`** тАФ Plugin manifest(s) consumed by Claude Code marketplace/install.
  - `plugin.json` тАФ Declares skills directory and agent file list; `version: 0.2.0`.
  - `marketplace.json` тАФ Marketplace registration metadata.
- **`rules/`** тАФ Always-on principles, loaded via SessionStart hook (not plugin auto-discovery).
  - `bootstrap.md` тАФ The single rule. Decision flow for "which skill to invoke", red flags, priority ordering (user > skill > default).
- **`skills/`** тАФ 12 user-triggered workflow skills, each a directory with `SKILL.md` (and optionally scripts/ancillary docs).
  - Pipeline (auto-routed): `mu-scope` тЖТ `mu-arch` тЖТ `mu-plan` тЖТ `mu-code` тЖТ `mu-review`
  - Orthogonal (auto-routed): `mu-explore`, `mu-debug`, `mu-retro`
  - On-demand (slash-only): `mu-biz`, `mu-prd`
  - Router: `mu-route`
  - Meta: `mu-write-skill`
- **`agents/`** тАФ Two dispatchable sub-agents with isolated context.
  - `mu-reviewer.md` тАФ Six modes: review-design, review-plan, review-code, review-compliance, review-coverage, review-security.
  - `mu-coder.md` тАФ Implementation specialist invoked by `mu-code`.
- **`knowledge/`** тАФ On-demand reference material, pulled via `@../../knowledge/...` relative paths.
  - `languages/` тАФ Per-language review criteria (java, go, python, typescript).
  - `templates/` тАФ Artifact templates (scope.md, explore.md, architecture.md).
  - `principles/` тАФ Thinking rubrics (10 files: inversion, premise-check, chesterton's-fence, git-safety, stance-detection, sign-off-gate, architecture-assessment, graphviz-conventions, skill-cso, skill-testing).
  - `reviews/` тАФ Review checklists (security-checklist, design-audit-rubric).
- **`hooks/`** тАФ Shell hooks registered via `hooks.json`.
  - `session-start` тАФ Reads `rules/bootstrap.md` and injects as `additionalContext` on SessionStart (startup/clear/compact).
  - `pre-tool-use/pipeline-gate.sh` тАФ Before Edit/Write, refuses unless `docs/scope/*.md` and `docs/specs/*-design*.md` both exist; exempts edits inside the plugin dir; fail-open.
  - `pre-tool-use/destructive-guard.sh` тАФ Warns on destructive Bash commands.
- **`docs/`** тАФ Human-facing documentation and artifacts generated by previous runs of the skills.
  - `architecture.md` / `architecture_cn.md` тАФ Canonical architecture description (EN/CN).
  - `testing.md` / `testing_cn.md` тАФ Testing approach.
  - `scope/`, `specs/`, `plans/`, `proposals/` тАФ Accumulated artifacts from running scope / arch / plan / proposal workflows on devmuse itself (dogfooding).
- **`tests/`** тАФ Organized as scenario directories, not a unit-test tree.
  - `brainstorm-server/`, `claude-code/`, `explicit-skill-requests/`, `hooks/`, `skill-triggering/`, `subagent-driven-dev/` тАФ Each holds harness/scenario material for validating skill-invocation behavior. Not auto-run as part of plugin install.
- **`.github/`** тАФ Issue template directory and PR template.
- **`README.md` / `README_CN.md`** тАФ Bilingual top-level docs.

## Entry Points

Execution is not "program start" тАФ it is Claude Code session start. Entry points:

- `hooks/session-start` тАФ Fires on every SessionStart (startup, clear, compact). Injects `rules/bootstrap.md` into the system context. This is THE boot path.
- `hooks/pre-tool-use/pipeline-gate.sh` тАФ Fires before every `Edit` or `Write`. Gatekeeps the pipeline (scope + design must exist).
- `hooks/pre-tool-use/destructive-guard.sh` тАФ Fires before every `Bash`. Destructive-command guard.
- User utterance тЖТ `Skill` tool invocation тАФ User types `/mu-scope`, `/mu-arch`, etc., or says something that triggers the bootstrap decision flow; Claude invokes the Skill tool on the matching `skills/<name>/SKILL.md`.
- `.claude-plugin/plugin.json` тАФ Static manifest read at plugin install time; declares agents explicitly and `skills/` for auto-discovery.

## Key Flows

### Flow: Session boot (always runs)
1. User opens Claude Code session (or runs `/clear`, `/compact`).
2. Claude Code runs `hooks/session-start`.
3. Script reads `rules/bootstrap.md`, JSON-escapes it, emits `hookSpecificOutput.additionalContext`.
4. Claude's system context now includes the bootstrap rule тАФ every subsequent user message is filtered through the "which skill applies?" decision flow.

### Flow: Feature-work pipeline (happy path)
1. User: "add feature X" (or similar).
2. Bootstrap rule fires тЖТ Claude invokes `mu-scope` skill.
3. `mu-scope` produces Use Case Set тЖТ writes `docs/scope/<date>-<slug>.md`.
4. Claude invokes `mu-arch` тЖТ writes `docs/specs/<date>-<slug>-design.md`; dispatches `mu-reviewer` in `review-design` mode.
5. Claude invokes `mu-plan` тЖТ detailed implementation plan; optional `mu-reviewer` `review-plan`.
6. Claude invokes `mu-code` тЖТ sets up worktree, dispatches `mu-coder` for implementation, then `mu-reviewer` `review-compliance` + `review-code` between tasks.
7. Claude invokes `mu-review` тЖТ dispatches `mu-reviewer` `review-coverage` (+ optional `review-security`), handles feedback, finishes with merge/PR/keep/discard.

### Flow: Pipeline gate enforcement
1. Claude attempts `Edit` or `Write` on a non-plugin file.
2. `pipeline-gate.sh` receives tool input JSON on stdin.
3. Extracts `file_path`. If path is inside `CLAUDE_PLUGIN_ROOT`, exits 0 (exempt тАФ enables meta-work on devmuse itself).
4. Else checks `docs/scope/*.md` exists; if not, emits `permissionDecision: deny` with message "Run mu-scope first."
5. Else checks `docs/specs/*-design*.md` exists; if not, emits deny with "Run mu-design first."
6. Both present тЖТ exit 0 (allow). Any error тЖТ `trap 'exit 0' ERR` fail-open (never blocks due to bugs).

### Flow: Bug fix (short-circuit pipeline)
1. User: "fix this bug".
2. Bootstrap тЖТ `mu-scope` with 1 use case = repro steps.
3. `mu-debug` (4-phase systematic root cause).
4. `mu-code` for the fix.
