# Scope: mu-wiki — Project Architecture Wiki Generation & Maintenance

> **Date:** 2026-04-29
> **Source:** User request — integrate deepwiki-style architecture documentation into DevMuse workflow

## Context

- DevMuse lacks a capability to generate and maintain project-level architecture documentation. mu-explore produces personal mental-model notes (shallow, session-scoped), and mu-arch only handles per-feature technical design. Neither produces a comprehensive, living architecture wiki.
- Inspired by [deepwiki-rs](https://github.com/sopaco/deepwiki-rs) (layered C4 multi-agent analysis) and [deepwiki-open](https://github.com/AsyncFuncAI/deepwiki-open) (two-phase structure→pages generation with source citations). After testing both tools on the DevMuse codebase, the deepwiki-open two-phase approach implemented via Claude Code subagents produced significantly better results than Litho's generic agents (no hallucination, accurate domain understanding, source citations).
- The wiki should integrate into the development workflow: generated when architecture understanding is needed, maintained incrementally, and consumed by mu-arch/mu-scope as context.

## Use Cases

### Happy Paths

- UC-1: When user executes `/mu-wiki generate` and `docs/wiki/_index.md` does not exist, Then execute two-phase generation (Phase 1: analyze file tree + README → output wiki structure JSON with sections, pages, relevant_files; Phase 2: parallel subagents generate each page as markdown with Mermaid diagrams + mandatory source citations), write output to `docs/wiki/`, user reviews → commit
- UC-2: When user executes `/mu-wiki update` and wiki exists, Then compare `_index.md` baseline commit with current HEAD via git diff, identify pages whose relevant_files have changed, regenerate only affected pages, user reviews → commit
- UC-3: When mu-scope Quick Probe determines risk=medium/high and `docs/wiki/_index.md` does not exist, Then suggest "建议先 `/mu-wiki generate` 生成架构 wiki", user can accept or skip, non-blocking
- UC-4: When mu-arch starts and `docs/wiki/` exists, Then read relevant wiki pages during "Explore project context" step as architecture context
- UC-5: When mu-arch design involves architecture-level changes (new components, module boundary changes, data flow changes) and wiki exists, Then before terminal (invoke mu-plan) suggest "设计涉及架构变更，建议 `/mu-wiki update`", user can accept or skip, non-blocking
- UC-6: When mu-scope Quick Probe determines risk=low, Then no wiki suggestion is triggered regardless of wiki existence

### Edge Cases

- UC-E1: Given wiki exists but baseline commit is >50 changed files behind HEAD, When user executes `/mu-wiki update`, Then warn "变更较大，建议 `generate` 重建而非增量更新" and let user choose
- UC-E2: Given project has >200k LOC, When user executes `/mu-wiki generate`, Then limit to top-level modules only (no deep dive), inform user they can later deep-dive specific subsystems
- UC-E3: When user executes `/mu-wiki generate` but wiki already exists, Then prompt "wiki 已存在，覆盖重建还是增量 update？"

### Error Cases

- UC-ERR1: When a page generation subagent fails (API error, timeout, etc.), Then other pages complete normally, failed page marked as `status: failed` in `_index.md`, user prompted to retry that page
- UC-ERR2: When `/mu-wiki update` cannot match git diff to relevant_files in `_index.md` (file renames, large-scale restructuring), Then degrade to suggesting full regenerate

### Reverse Cases (must NOT happen)

- UC-R1: When mu-explore executes any variant (onboarding, takeover, pre-change, pre-debug, dep-eval), Then its behavior, output format, and output path (`docs/explore/`) must remain completely unchanged
- UC-R2: When mu-scope or mu-arch suggests wiki but user skips, Then each skill continues its normal flow without blocking or degrading
- UC-R3: When `docs/wiki/` is created, Then existing `docs/explore/` artifacts are not affected

## Conflicts

- ⚠️ REFINEMENT: UC-3 vs UC-6 — initial UC-3 trigger was "cross-module fan-out" which would over-trigger on cross-module bug fixes. Refined to "risk=medium/high" which naturally excludes low-risk cross-module tasks.
  - Resolution: UC-3 condition is `risk=medium/high + no wiki`, not just `cross-module + no wiki`

## Non-Functional Constraints

- [Performance] Phase 1 (structure generation) must complete in a single subagent call (<2 min). Phase 2 (page generation) should parallelize subagents where possible.
- [Token efficiency] Each page generation subagent scoped to that page's relevant_files only — no full-repo context injection.
- [Anti-hallucination] Every page must include source citations (`Sources: [file:lines]()`) and a `<details>` block listing all referenced files. Information must come solely from source files.

## Constraints & Assumptions

- mu-wiki is an **on-demand skill** (like mu-biz, mu-prd) — not auto-routed by mu-route
- Wiki output is markdown files in `docs/wiki/` — no external tools, databases, or embedding pipelines required
- `_index.md` serves as both navigation index and tracking metadata (baseline commit, per-page relevant_files, page status)
- Claude Code subagent mechanism is the execution engine for both phases — no external CLI tools (Litho/deepwiki) needed
- Assumes git is available for diff detection in update flow

## Out of Scope

- mu-explore changes — mu-explore remains unchanged, wiki is a separate skill
- Auto-routing via mu-route — mu-wiki is on-demand only
- Web UI / interactive wiki viewer — output is plain markdown files
- Embedding/RAG pipeline — context is provided via direct file reading, not vector search
- Automatic wiki update hooks — updates are user-initiated (via `/mu-wiki update` or accepting a suggestion)

## Impact Analysis

- Affected modules: mu-scope (1 suggestion line in Quick Probe), mu-arch (1 read step + 1 suggestion line at terminal)
- Existing tests that may break: None (new capability, minimal changes to existing skills)
- Migration needs: No — purely additive
