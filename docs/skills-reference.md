# Skills Reference / 技能参考手册

## Workflow Overview / 工作流总览

```
craft-brainstorm → craft-plan → craft-sdd / craft-execute → craft-finish
       │                              │
       │                        craft-tdd (每个任务)
       │                        craft-review (每个任务后)
       │                        craft-debug (遇到问题时)
       │                        craft-verify (完成前)
       │
  craft-worktree (创建隔离工作区)
```

---

## Skills List / 技能清单

### craft-brainstorm — 头脑风暴

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-brainstorm` |
| **触发时机** | 任何创造性工作之前：新功能、新组件、行为修改 |
| **作用** | 通过苏格拉底式对话，将粗略想法细化为完整设计方案 |
| **工作流** | 探索上下文 → 逐一提问 → 提出 2-3 个方案 → 分段展示设计 → 写设计文档 → 规格审查 → 用户确认 → 交接给 craft-plan |
| **输出** | 设计文档保存到 `docs/craft-claude/specs/` |
| **依赖** | 终态调用 craft-plan |

---

### craft-plan — 编写计划

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-plan` |
| **触发时机** | 设计方案获批后，写代码之前 |
| **作用** | 将设计拆分为详细的实施计划，假设执行者零上下文 |
| **工作流** | 审查范围 → 规划文件结构 → 定义小任务（每个 2-5 分钟） → 每个任务包含：文件路径、失败测试、实现、验证、提交 |
| **输出** | 计划文档，包含可执行的任务列表 |
| **依赖** | 需要 craft-brainstorm 完成；执行前需 craft-worktree |

---

### craft-sdd — 子 Agent 驱动开发

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-sdd` |
| **触发时机** | 有计划后，当前会话中执行 |
| **作用** | 为每个任务派遣独立子 Agent，两阶段审查确保质量 |
| **工作流** | 读取计划 → 逐任务：派遣实现者 → 规格符合性审查 → 代码质量审查 → 标记完成 → 最终全局审查 → craft-finish |
| **优势** | 上下文隔离，质量高于 craft-execute |
| **依赖** | 需要 craft-plan；结束调用 craft-finish |

---

### craft-execute — 执行计划

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-execute` |
| **触发时机** | 有计划后，无子 Agent 或独立会话中执行 |
| **作用** | 按顺序执行计划中的所有任务 |
| **工作流** | 加载计划 → 批判性审查 → 逐任务执行 → craft-finish |
| **注意** | 如果有子 Agent 可用，优先使用 craft-sdd |
| **依赖** | 需要 craft-plan；需要 craft-worktree；结束调用 craft-finish |

---

### craft-parallel — 并行 Agent 调度

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-parallel` |
| **触发时机** | 面临 2+ 个独立任务，无共享状态或顺序依赖 |
| **作用** | 每个独立问题域派遣一个 Agent 并发工作 |
| **工作流** | 识别独立域 → 创建聚焦的 Agent 任务 → 并行派遣 → 审查整合结果 |
| **适用场景** | 多个不同子系统的测试失败、不相关的 bug |

---

### craft-worktree — Git 工作树

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-worktree` |
| **触发时机** | 开始功能开发前，执行计划前 |
| **作用** | 创建隔离的 git worktree，防止在 main 上直接工作 |
| **工作流** | 检查现有目录 → 检查 CLAUDE.md 偏好 → 创建 worktree 和新分支 → 切换到工作区 |
| **依赖** | 需要 git 仓库 |

---

### craft-finish — 完成开发分支

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-finish` |
| **触发时机** | 所有任务完成，测试通过 |
| **作用** | 引导完成流程：验证、选择整合方式、清理 |
| **工作流** | 验证测试 → 确定基础分支 → 展示 4 个选项（本地合并 / 推送+PR / 保留 / 丢弃）→ 执行选择 → 清理 worktree |

---

### craft-tdd — 测试驱动开发

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-tdd` |
| **触发时机** | 实现任何功能或修复 bug 前 |
| **作用** | 强制 RED-GREEN-REFACTOR 循环 |
| **工作流** | RED：写失败测试 → 确认失败 → GREEN：写最少代码通过 → 确认通过 → REFACTOR：保持绿灯下重构 |
| **铁律** | 没有失败测试就不写生产代码。先写代码再补测试？删掉重来 |

---

### craft-debug — 系统化调试

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-debug` |
| **触发时机** | 遇到任何 bug、测试失败、意外行为 |
| **作用** | 先找根因再修复，防止掩盖症状 |
| **工作流** | 阶段 1：根因调查（仔细读错误、复现、检查变更） → 阶段 2：假设形成 → 阶段 3：定向测试 → 阶段 4：实现修复并验证 |
| **铁律** | 没有根因调查就不修复 |
| **附带工具** | `root-cause-tracing.md`、`defense-in-depth.md`、`condition-based-waiting.md`、`find-polluter.sh` |

---

### craft-review — 请求代码审查

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-review` |
| **触发时机** | 完成任务后、重要功能实现后、合并前 |
| **作用** | 派遣独立审查子 Agent，捕获问题避免级联 |
| **工作流** | 获取 git SHA → 派遣 craft-reviewer 子 Agent → 接收反馈 → 按严重程度处理（Critical → Important → Minor） |

---

### craft-review-response — 接收代码审查

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-review-response` |
| **触发时机** | 收到代码审查反馈后，实施建议前 |
| **作用** | 技术性评估反馈，而非盲目同意 |
| **工作流** | 通读完整反馈 → 用自己的话复述需求 → 对照代码验证 → 评估技术合理性 → 技术性回应或反驳 → 逐项实施并测试 |

---

### craft-verify — 完成前验证

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-verify` |
| **触发时机** | 声明工作完成前、提交/PR 前 |
| **作用** | 用实际证据验证，防止虚假的完成声明 |
| **工作流** | 确定验证命令 → 运行新鲜命令 → 读取完整输出和退出码 → 确认输出支持声明 → 然后才声明完成 |
| **铁律** | 没有新鲜验证证据就不声明完成 |

---

### craft-write-skill — 编写技能

| 项目 | 内容 |
|------|------|
| **调用** | `/craft-write-skill` |
| **触发时机** | 创建新技能、编辑现有技能 |
| **作用** | 将 TDD 方法论应用于流程文档编写 |
| **工作流** | RED：运行压力场景，观察 Agent 无技能时的行为 → GREEN：写最小技能文档 → REFACTOR：堵住新发现的合理化漏洞 |
| **铁律** | 没有失败的基线测试就不写技能 |
| **依赖** | 需要理解 craft-tdd |
